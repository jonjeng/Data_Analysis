Preparing for a MongoDB Database¶

We'll now prepare the dataset for a MongoDB database. We must:

  • address potentially mispelled street types (e.g., if the same concept is spelled different ways, we won't see that the different results refer to the same concept)
  • convert our XML file to JSON, as MongoDB databases don't "understand" the former. During this process, we'll address data that cannot be included in the database</a>

Let's begin with mispelled data. The most obvious issue is that of street names. Some may abbreviate street names, with or without punctuation, whereas others may not. Thus, we may have:

  • Market Street
  • Market St.
  • Market St,
  • Market St
  • and etc.

These may all refer to the same entity but if we regard different names as referring to different entities, we will end up thinking that there are several different and less prominent entities when there is in fact, a single, relatively prominent entity.

Let's address that issue. Our first step is to create a dictionary containing as keys all alternate spellings of a name. At first glance, possible names should be:

  • Street,
  • Avenue,
  • Boulevard,
  • Drive,
  • Court,
  • Place,
  • Square,
  • Lane,
  • Road,
  • Trail,
  • Parkway, and
  • Commons

We'll put all of those names in a list, expected and compile a regular expression object to match the last token in a string since we expect street names to end with a road name, as in expected and to possibly be followed by a period (e.g., if abbreviated or at the end of a sentence). Since capitalization differences may occur, we must stipulate that the case of words not be considered. To get a list of all alternative spellings of street names, we'll take the final token, using the regular expression object, of each street name. If that name is not in our list of expected street names, we'll add the particular value (i.e., entire street name) to a dictionary called street_types. The keys of the dictionary will be all street types (e.g., St., Ave., Blvd.) and the values will be all of the corresponding entire street names.

CREATED = [ "version", "changeset", "timestamp", "user", "uid"]

def shape_element(element):
    
    if element.tag == "node" or element.tag == "way" :
        # Each element will be represented as a dictionary with 'id' and 'type' keys
        #  and a dictionary of information about when the entry was created
        node = {
            'id'      : element.attrib['id'],
            'type'    : element.tag,
            'created' : {}
        }
        
        # Add the information about when the entry was created
        for attr in CREATED:
            if attr in element.attrib:
                node['created'][attr] = element.attrib[attr]
        
        # Add the position information accordingly
        if 'lat' in element.attrib:
            node['pos'] = [float(element.attrib['lat']), float(element.attrib['lon'])]
        
        # Make a list of other information to add. We've already added
        #  information about the entry creation and position, so remove
        #  those from the list
        attribs = list(element.attrib)
        for i in CREATED:
            if i in attribs:
                attribs.remove(i)
        if 'lat' in attribs:
            attribs.remove('lat')
            attribs.remove('lon')
        
        # Add the information we haven't yet added
        for attr in attribs:
            node[attr] = element.attrib[attr]
        
        # Now add information from second level elements. Regardless of the type
        #  of the top  level element, we'll want to add information from second
        #  level elements with the tag 'tag'.
        for tag in element.iter('tag'):
            if tag.tag == 'tag':
                k = tag.attrib['k']
                
                # If the k value is problematic, skip it. If it begins with addr:,
                #  assume it only consists of two "levels" and proceed accordingly. 
                #  Otherwise, if it consists of two other levels, add the information
                #  accordingly. If it consists of more than two "levels", skip it.
                #  In every other case, add the information straightforwardly
                if re.search(problemchars, k):
                    continue
                
                elif k[:5] == 'addr:':
                    if ':' not in k[5:]:
                        if 'address' not in node:
                            node['address'] = {}
                        k_split = k.split(':')
                        name = tag.attrib['v'].title()
                        if k_split[1] == 'street':
                            m = street_type_re.search(name)
                            if m:
                                street_type = m.group()
                                if street_type in mapping:
                                    if len(name.split()) <= 5:
                                        name = name.replace(street_type, mapping[street_type])
                        node['address'][k_split[1].lower()] = name
                
                elif ':' in k:
                    k_split = k.split(':')
                    if len(k_split) > 2:
                        continue
                    node[k_split[0].lower()] = k_split[1].title()
                
                else:
                    if k.lower() != 'address':
                        node[k.lower()] = tag.attrib['v']
                    else:
                        split_address = tag.attrib['v'].split()
                        if len(split_address) == 1:
                            continue

                        node['address'] = {}
                        node['address']['housenumber'] = split_address[0]
                        node['address']['street'] = ' '.join(split_address[1:])
        
        # If the top level element is a way, we'll add information from second level
        #  elements with the tag 'nd'
        if element.tag == 'way':
            node['node_refs'] = []
            for tag in element.iter('nd'):
                node['node_refs'].append(tag.attrib['ref'])
        
        # Finally, return the result
        return node
    
    else:
        # If the top level tag is neitner 'node' nor 'way', do nothing
        return None

Conclusions and Other Ideas About the Dataset¶

We specifically examined OpenStreetMap data corresponding to a dataset labelled on Metro Extracts as San Francisco, CA. The data was bounded by a rectangular region, as in the line <bounds minlat="37.449" minlon="-122.737" maxlat="37.955" maxlon="-122.011"/>. The dataset was quite large (1.414 GB as an uncompressed XML) and unlikely to work with our memory constraints, so a MongoDB database was used. To load the data into a MongoDB database, though, we had to convert the file to a JSON format.

In the process, we addressed potentially mispelled street types and data that cannot be included in a MongoDB database. The data finally consisted of a list of dictionaries, which conformed to the JSON format and were stored in a separate file approximately 2.0 GB in size.

We then proceeded to examine the data with the help of MongoDB. We found that:

  • a very small proportion, about 0.00171%, of element types were not listed correctly
  • the users that contributed most and that the distribution of contributions is sharply skewed, such that less than 1% of the users made over 80% of the contributions to the dataset
  • if we assume regions based on clusters, each the various regions the dataset encapsulates seem fairly well represented. We see five regions, as may be expected, and each has a reasonable number of elements.
  • in the most represented cities, restaurants are the most commonly reported ammenity
  • "En" and "Zh" are by far the most commonly reported restaurants. This may be because the names appear to be a catch-all of sorts for English and Chinese restaurants, respectively, and suggests that the numbers may not be as informative as we'd like
  • several banks were spelled inconsistently. This seems to reflect a shortcoming in data collection
  • The vast majority of places of worship are categorized as Christian. There aren't many other religions represented but spelling does not appear to be an issue here

Other Ideas About the Dataset¶

It is encouraging to see that the different portions of the area studied appear well represented. For the time being, it appears business decisions may be shaped, in part, by this data.

In the future, alternative spellings may be made more consistent and more far-reaching conclusions can be drawn about the prevalence of certain establishments. Further, discontinued establishments (e.g., a store that closed) may be flagged for removal and subsequently removed from the dataset. Users could report on the continuity of establishments and infrastructure.

Considering the success of Wikipedia, similar ideas could be adopted for the OpenStreetMaps dataset. Data could be made more easily evaluated for accuracy by other users and users could be allowed to edit contributions of other users. With regard to editing, a system of repute and/or credibility could be established to ensure that data is more likely to be improved in terms of accuracy than to be corrupted by pranksters. Alternatively or in addition, edits could be required to be checked by other a particular number parties, such as other users.

It is possible that even with such an advent, the room for inaccuracies, such as from dishonest tampering, may still be too great. For instance, an incredible trove of expert-level information may be found on Wikipedia, and inaccuracies, while not very prevalent, still exist. It remains to be seen, though, whether an analog of such inaccuracies in OpenStreetMaps is acceptable.

It may also be advisable to make contributing an easier process. As it stands, the number of contributors may not be inspiring - we found less than 3,000 unique users contributed to this particular dataset, which does not compare well with the population of 7 million people in the Bay Area (source). Even if you assume that over half of the population is too busy, too young or too old to either think at a sufficiently advanced level or operate a computing device, or unable to contribute anything of value, there still remains a large, untapped base of potential users.

This suggests that the process of contributing is excessively unpalatable. If the process is too difficult (e.g., requiring one to peruse a lengthy beginner's tutorial) and/or time consuming, such should be addressed. For instance, the prevalence of smartphones could be taken advantage of. Data contribution could be made far easier by means of a smartphone application and users could opt to have their smartphones automatically provide information, thereby providing further details such as population density and the degree to which a road is utilized or an establishment is frequented, among other details.

In addition, incentives may be provided for contributing data. The use of reCAPTCHA to collect data for the digitization of books comes to mind. The existence of a such a necessary yet simple task that could serve as an avenue for the collection of data for OpenStreetMaps is not intuitive. Perhaps we will never find a scenario as convenient as distinguishing human activity from automated activity (i.e., that which reCAPTCHA capitalizes on), but it is possible that a source of incentives (e.g., the provision of personal satisfaction) could be discovered and utilized to further the complexity and accuracy of OpenStreetMap data.

As a result of these efforts, it is possible that the results of the analysis may be different after such is done. It is possible that more reliable and complex business decisions could be made on the basis of analyses of this dataset. These anticipated issues and potential benefits should be analyzed by influential parties.
